# Albedo（炼真）立项调研报告

> 日期：2026-07-09
> 定位：五器工坊之「校验」环节 —— 对 Nigredo 采集来的一人公司经验，做**质量评估 + 优点分析 + 炼真提纯**，输出可信、可模仿、结构化的经验，喂给 Citrinitas。
> 本文档为 Phase 0.5 探索产出，供 Phase 1 拆解前确认范围。

## 一、调研结论摘要

1. **GitHub 上"蒸馏别人经验"的赛道 2026 年已成形**，最相关的开源项目是 **pangu-skill（盘古）**——它把人物/领域公开知识蒸馏成可调用 Skill，流程里**已经包含可信度审计 + 优点提取**。这证明我们的方向成立，但它是**端到端单人项目**，而我们已有 Nigredo/Citrinitas/Rubedo 分工，Albedo 只取其「提炼 + 质检 + 优点萃取」中段，不复制全链路。
2. **「优点分析」不是空概念**，pangu 的"六层提取结构 + 领域协议"给出了具体可落地的子能力清单（见第三节）。
3. **除质量评估 + 优点分析外，还能做 7 类事**（结构化提炼、矛盾检测、溯源、冲突仲裁、适用性评估、时效判定、知识图谱关联），其中"跨源矛盾检测"已有学术方案（arxiv 2504.00180 + IUQ 原子主张法）。
4. **建议 MVP 先做最小闭环**（质量评估 + 优点分析 + 结构化 + 溯源），跨源矛盾检测等规模上来再做——符合你"规模到了再说"的一贯原则。

## 三、「优点分析」可以做什么

"优点分析"= 从一条一人公司经验里，提取**值得模仿、可迁移**的部分。基于 pangu 的"六层提取结构 + 领域协议"，拆成 6 个具体子能力：

| 子能力 | 说明 | 对应你的需求 |
|---|---|---|
| 1. 核心洞察提取 | 一句话点出他做对的本质（如"第一性原理""聚焦核心场景"） | 抓住可复用的方法论内核 |
| 2. 可复用 SOP 步骤 | 把经验拆成可照搬的动作序列 | **直接支撑你"多 SOP 并列"** |
| 3. 差异化亮点 | 相比普遍做法，他独特在哪（反模式对照） | 找出真正有价值的打法 |
| 4. 适用场景匹配 | 这套打法适合什么阶段/什么生意 | 对你：酷家乐/小红书/工具付费/B站 分别适用度 |
| 5. 陷阱预警 | 他踩过的坑 / 别人容易误用之处 | pangu 的"诚实边界"，防你照搬翻车 |
| 6. 迁移成本评估 | 照搬这套需要什么前置条件（资源/技能/阶段） | 判断"我现在能不能抄" |

## 四、除了"质量评估 + 优点分析"，Albedo 还能做什么

按研究与 pangu 流程，列出 7 个候选能力（标注建议纳入阶段）：

| 候选能力 | 说明 | 建议阶段 |
|---|---|---|
| A. 结构化提炼（Structuring） | 把混乱的视频字幕 → 结构化 SOP / 知识树 | **MVP**（"多 SOP 并列"的载体） |
| B. 跨源矛盾检测（Contradiction） | 多个教程对同一件事说法不一 → 标记冲突 | 规模期（见第五节技术参考） |
| C. 溯源（Provenance） | 这条知识来自哪个视频/UP主/时间戳 | **MVP**（防编造式交付） |
| D. 冲突仲裁建议（Arbitration） | 矛盾时给"信谁"的建议（基于来源可信度/一致性/时效） | 规模期（依赖 B） |
| E. 适用性评估（Applicability） | 对你具体业务线的适配度打分 | MVP+（接你 4 条业务线） |
| F. 时效判定（Recency） | 这个方法是否过时（B站很多旧打法） | MVP+ |
| G. 知识图谱关联（Linking） | 和 Citrinitas 已有知识连边 | 远期（为 Citrinitas v1.4.0 知识关系网铺路） |

## 五、矛盾检测的技术参考

- **论文 arxiv 2504.00180《Contradiction Detection in RAG Systems》**：解决 RAG 检索到矛盾信息导致输出不一致的问题，直接对应"知识库内/新摄入 vs 已有知识"的冲突检测。
- **IUQ 方法**：把生成内容拆成**原子主张（atomic claim）**，用 NLI 模型基于给定上下文做三元评估（一致/矛盾/无关），量化与可靠信源的不一致程度。
- **启示**：Albedo 的矛盾检测不必从零造——`原子主张拆分 + NLI 一致性比对 + 跨源投票`是一条已验证的路径。规模期接入即可。

## 六、架构师视角：Albedo 在五器中的定位

```
Nigredo(采集) ──生料──▶ Albedo(炼真) ──炼真后经验──▶ Citrinitas(存储) ──▶ Rubedo(变现)
                         ↑提炼+质检+优点萃取
                         ✗ 不采集 ✗ 不存储 ✗ 不变现
```

- **Albedo 的本质 = 「炼真」**：对生料做提炼 + 质检 + 优点萃取，产出"炼真的、可信的、带优点的结构化经验"。
- **不复制 pangu 全链路**：pangu 是端到端单人项目；我们已有分工，Albedo 只做中间炼真段。
- **开发形态**：独立项目 `D:\albedo`（与 Nigredo/Citrinitas/Rubedo 并行节奏一致）；接口严格对齐 `contradiction_report` schema；未来五器耦合时按模块化单体集成进 Opus Magnum。

## 九、盘古（pangu）能力 vs 五器覆盖缺口分析

将 pangu 的 8 项核心能力逐一映射到五器分工：

| pangu 能力 | 对应项目 | 覆盖？ |
|---|---|---|
| 多源采集（六路并行） | Nigredo（采集） | ✅ |
| 双协议提炼（人物/领域） | Albedo（炼真·提炼） | ✅ |
| 质量验证与审计 | Albedo（质量评估） | ✅ |
| 批评者视角采集 | Nigredo（多源）+ Albedo（用于可信度） | ✅ |
| 诚实边界标注 | Albedo（优点分析·陷阱预警） | ✅ |

| **开天重写（伪前提校正）** | —— | ⚠️ **缺口**：无项目负责"用户伪前提输入先重写" |
| **产出可安装 Skill 仓库** | —— | ⚠️ **缺口**：精炼结果→可安装/可执行 skill 的产品化封装，介于 Albedo 与 Rubedo 之间，无明确归属 |

- **产出可安装 Skill 产品化** → 建议归 **Rubedo（变现·SOP 建立）**：Albedo 产出精炼经验，Rubedo 把它变成可执行 SOP/skill（Rubedo v0.3.0 酷家乐 SOP 正是此职责）。缺口因此闭合。

→ 你的架构是完整的，只需把这两项写进对应项目边界。

### 11.5 竞品共性 & Albedo 的差异化空间（架构师判断）

1. **所有竞品都是"端到端单人工具"**——自己采集、自己提炼、自己存、自己产出 skill。Albedo 是五器分工里的"炼真中段"，只做认知精炼，不碰采集/存储/变现。这是本质差异，也避免重造轮子。
2. **验证机制是共同强项**：nuwa 的三重验证 + 保真度评分卡、OpenFactCheck 的统一核查管线——Albedo 质量评估模块应直接借鉴，不自研。
3. **商业化产品已验证"视频→SOP"有付费需求**（TubeScribed），且 SOP 格式标准化——Albedo 结构化提炼产出对齐该格式即可无缝进熔知/Rubedo。
4. **真正的空白（Albedo 的未来护城河）**：
   - **跨源矛盾检测**：竞品要么单人蒸馏（无矛盾），要么单条事实核查（不跨源比对经验）。Albedo 未来规模期做"多教程互相印证/冲突"是独特空间。
   - **适用性评估（对你 4 条业务线适配度）**：所有竞品是通用人物蒸馏，不为特定业务线（酷家乐/小红书/工具付费/B站）做适配打分。这是 Albedo 对“一人公司”场景的独特价值。（📝 补注：此适用性评估/业务线适配度评分已于 2026-07-09 从 Albedo 移出，归 Rubedo/OpusMagnum；Albedo 不再做业务线适配，详见 PROJECT_PLAN.md 版本路线注。）
