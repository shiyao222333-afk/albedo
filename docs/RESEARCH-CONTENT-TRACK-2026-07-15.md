## 0. 本文定位与范围

**Track A 内容线** = 处理"视频**讲了什么**"：知识点、核心洞察、可照搬的 SOP、陷阱坑点、以及真伪验真。
- **锚 = 字幕**（在前），AI 生成的简介做辅助钥匙（用户已定调）。
- 与 **Track B 形式线**（逻辑/结构/表达/制作手法）严格正交，两者各一套工具和流程（见 TWOTRACKS 调研）。本文只谈 A。

**数据流前提**：Nigredo 已吐出"单文件中转"（字幕 + AI 简介 + 高光时间点 + 高光前后弹幕 + 高赞/置顶评论 + 播放分析）。内容线必须把这些信号**组合**分析，字幕是核心，其它是分析的钥匙。

**蓝图映射**（Albedo 宪法）：内容线覆盖蓝图的"优点分析（核心洞察/SOP/陷阱）+ 结构化提炼 + 溯源 + 验真 + 内容净化"。形式线覆盖不了这些。

## 1. 角度一：方法论——内容分析该用什么框架

### 1.1 主题分析 vs 内容分析

| 维度 | 主题分析 Thematic Analysis | 内容分析 Content Analysis |
|------|------|------|
| 关注点 | 跨数据集的"模式化含义" | 意义单位的"频率和分布" |
| 典型分析单元 | 片段 /  episode / 整段 | 词 / 短语 / 句子 |
| 产出 | 解释现象的主题（叙述式） | 带计数的分层类别（可喂看板） |
| 计数角色 | 可选、次要 | 通用、支撑比较 |
| 抽象层级 | 表面→潜在 | 更接近表面 |
| 适用 | "为什么/怎么做" | "是什么/多少次" |

来源：delvetool / kopia.ai / nvivo / articos 对比文。

**对我们有什么用**：内容线两样都要，但阶段不同——
- **主题分析**（宽角）：先搞清"这条视频整体在讲哪几个主题"，给报告定骨架。
- **内容分析**（可计数）：优点/SOP/陷阱各自出现几次、哪些被高赞评论印证——这些计数进看板，也支撑"稳定性"评估。

### 1.2 固定码本（Codebook）+ 演绎式编码 = 治"输出不稳定"的核心

学术标准做法（quallmer / ar5iv 2306.00176 / themoonlight codebook-reduction）：
1. **先定义码本**：类别名 + 简短定义 + 正例/反例。码本是 LLM 的 prompt 指令。
2. **逐段编码**：把数据切成小段，每段套码本打标。
3. **验证一致性**：两个人类专家 + LLM 用**同一码本**标注同一批样本，算 Krippendorff's alpha / Cohen's kappa。低就改码本再测。

**关键洞察**：码本把"可复现性"从"模型努力"变成"约束"。这正是用户另一个任务（**相同输入输出不稳定**）的解法之一——问题不在模型，在于我们没给固定码本。

**对我们的直接意义**：内容线的输出结构由蓝图里的固定框架（核心洞察 / SOP / 陷阱 / 事实主张）决定，**不是每次自由发挥**。

### 1.3 熟悉化（Familiarization）不能跳

Braun & Clarke 六阶段第一步就是"通读数据、写备忘录"。用户上一轮已点过"分析太肤浅"——根因常是跳过熟悉化直接编码。
**→ 内容线第一步必须是整文件通读字幕 + 简介（EDA，不急着出结论），再编码。**

## 2. 角度二：价值萃取——怎么把视频变成"可照搬的知识"

### 2.1 SOP / 操作步骤抽取（竞品怎么做）

**手工法**（tagplan.app，质量经理实战）：
- 铁律：**写"做了什么"不是"说了什么"**。老手口述会跳过"显然"的步骤，视频全抓到了。
- 四段式结构：Preparation（工具/材料/PPE）→ Main（编号步骤，一步一动作）→ Verification（质检点）→ Completion（清理/交接）。
- 五类 callout：DANGER / WARNING / CAUTION / QUALITY / NOTE。
- 用**新人模拟**抓 80% 文档错误（歧义工具名、缺分支逻辑、缺验收标准）。

**AI 法**（Scribe / ScreenApp / tinyteam）：
- 录屏 → OCR 读屏上字 → 步骤识别 → 编号 SOP + 截图。
- 黄金 UX 模式：**capture-first, write-second**（先做再写，比"描述你做了什么"转化率高得多）。AI 写 80% 描述，人改比从头写快。
- Scribe 靠这个做到 $30M ARR——"文档是没人想交的税"，它直接免掉写作。

**对我们有什么不同（关键）**：我们不是录自己屏，是**读别人视频的字幕**。字幕是"说了什么"不是"做了什么"。
**→ 内容线内部需要第二层分流**：
- **教程型视频**（有操作步骤可抽）→ 抽 SOP。
- **观点型视频**（无 SOP，只有论点/方法）→ 抽核心洞察 / 论点，不硬抽步骤。

### 2.2 优点 / 洞察 / 陷阱 三元组

蓝图定义"优点分析 = 核心洞察 / SOP / 陷阱"。这其实就是固定码本的**三个主码**：
- 核心洞察（论点 / 方法 / 心法）
- SOP（可照搬步骤）
- 陷阱（反例 / 坑 / 别踩）

**竞品空白**：几乎所有摘要/笔记工具都不专门抽"陷阱/坑点"——这是我们的差异化切口。

## 3. 角度三：验真——怎么判断"真 / 假 / 可疑"

### 3.1 通用事实核查流水线（OpenFactCheck / FacTool）

**OpenFactCheck**（开源框架，三模块）：
- RESPONSEEVAL：把"文档拆主张 → 检索证据 → 判定真伪"固化为可插拔流水线。
- 三件套类：`claim_processor`（拆主张）/ `retriever`（取证据）/ `verifier`（判真伪）。
- YAML 可配，模块可混搭（RARR / FacTool / FactCheckGPT 的部件任意组合），从任意步起。

**FacTool**（工具增强框架）：
- 4 类任务：知识 QA / 代码 / 数学 / 科学文献。
- 主张级 + 响应级事实性评分；输出含**证据 URL + 修正建议**（不直接改写，只标错）。
- 学术标杆数据：claim-level accuracy（主张级）GPT-4=75.6%，response-level（整段都对）=43.3%。
  **→ 启示："整段都说对"极难，必须拆到主张级逐个验。**

### 3.2 接女娲（nuwa）三重验证的位置

工作记忆：女娲 = 验真对标项目。内容线把"抽出的主张"交给 nuwa 做三重验证，正好对应 claim_processor（抽主张）/ retriever（取证据）/ verifier（判真伪）。

### 3.3 Source-grounded（以来源为锚）减少幻觉

**NotebookLM 哲学**：回答严格锁在上传来源，每句带引用，幻觉大幅下降；它"不发散"反而是卖点。
**→ 我们的溯源天然契合**：每条洞察 / 主张必须能点回**字幕时间戳 + 原视频链接**。
**验真不是"上网查证全部"**，而是"主张 → 证据 → 可信度"三级，可信度聚合（FPF）进熔知。

## 4. 角度四：稳定性——怎么让相同输入输出相同

（直接服务用户的第二个任务：输出不稳定）

### 4.1 受限解码 / JSON Schema / Structured Outputs

机制（cheungto / tensoria / acingai / devstarsj）：
- 推理时约束 token，非法 token 概率置 -∞；模型**"不能"违规，不是"更努力"**。
- 可靠性排序：**固定 choice（最稳）> regex > 扁平 JSON > 深嵌套（最不稳）**。
- 主流厂：OpenAI Structured Outputs（复杂 schema 遵循 100%）、Anthropic tool use、Gemini response_schema、本地 Ollama 用 xgrammar。

**对我们**：
- 内容线抽取用 **Pydantic schema + enum**（优点/SOP/陷阱固定枚举）+ 扁平两级嵌套。
- **temperature = 0**。

### 4.2 LLM-as-a-Judge：评分也要确定性

四维 prompt（galtea）：
1. 标准定义（用领域词，不写"高质量"这种空话）
2. 推理结构（逐主张枚举，不是整体印象）
3. 评分规则（确定性映射到 verdict，去掉"用你的判断"）
4. 边界处理（截断/空检索/拒答怎么算）

**要点**：一个维度一次调用；fine-grained 5 分不如 binary 稳；越精细分越招噪声。

**对我们**：质量评估（真/假/可疑）用**枚举 + 单维度调用**。

### 4.3 综合：任务二的解法

内容线输出不稳定，根因是"自由格式 + 高温 + 无码本"。
**解法 = 固定码本（洞察/SOP/陷阱/主张）→ schema 枚举约束 → temperature 0 → 单维度评分。**
这与任务一（馏析数据处理）同源——一次改好，两个病都治。

## 5. 角度五：组合分析——字幕 + 高光 + 弹幕 + 评论怎么搅在一起

### 5.1 时间轴对齐（字幕为锚）

- **CSDN 实战工作流**：字幕 + 评论按时间戳映射，`context_summary` 取字幕前后各 1 句；偏差 >5 秒的 0 条——**可验证性是整套方案的生命线**。
- **YouTubeSenseAI / crowdlisten**：评论聚类成主题、按互动加权，**链接回"创作者实际说了什么"**；"这条评论说你错了"→ 跳到对应字幕句。

### 5.2 高光 = 注意力锚点

- 高光时间点（Nigredo 已抓）是"观众最兴奋 / 最有争议"的位置。
- 内容线：**取高光前后各 N 秒字幕 + 该窗口弹幕 → 组合成"高光上下文块"**，优先深度分析。
- 弹幕在高光处密度最高，是观众正负反馈的浓缩小料（多模态时空失配的解法 = 语义窗口 + 跨模态锚点，见首篇调研）。

### 5.3 评论路由

- 高赞评论（已按 LIKE 排序）、置顶评论（作者定调）作为内容线"观众侧印证 / 反驳"。
- 评论若说"学到了"→ 内容线；说"剪辑丝滑"→ 形式线；说"没懂"→ 内容线 + 可能需形式线辅助。逐条判 A/B（见 TWOTRACKS）。

## 6. 竞品全景——他们怎么处理"内容"

| 工具 | 类型 | 内容处理做法 | 验真 | SOP抽取 | 多源组合 | 对我们的启示 |
|------|------|------|:--:|:--:|:--:|------|
| **TubeScribe** | 本地视频→文档 | 字幕→结构化MD+评论情感+高赞评论 | ❌ | ❌ | 字幕+评论 | 隐私/本地思路可借鉴；无验真无SOP |
| **NoteGPT / Glasp** | 摘要/笔记 | 摘要+思维导图+Notion同步 | ❌ | ❌ | 单字幕 | 结构化笔记成熟，但浅 |
| **通义听悟 / 飞书妙记 / 秘塔** | 国内转写 | 转写+摘要+章节+关键词 | ❌ | ❌ | 单音频 | 生态强，无验真无SOP抽取 |
| **Get笔记 / Kimi** | 长视频笔记 | 音视频→笔记+知识库问答（豆包） | ❌ | ❌ | 单源 | 知识库问答好，无验真 |
| **NotebookLM** | 来源锚定 | 多源综合+引用，不发散 | ❌（不主动验） | ❌ | 多文档综合 | **source-grounded 哲学必学** |
| **OpenFactCheck / FacTool / RARR** | 验真框架 | 文本主张拆验 | ✅ | ❌ | 仅文本 | **验真流水线直接复用** |
| **Scribe / ScreenApp** | SOP生成 | 录屏/OCR→编号SOP | ❌ | ✅ | 单录屏 | capture-first 模式；但针对"自己做的" |
| **Video Mood / YouTubeSenseAI / crowdlisten** | 视频+评论 | 内容+社区反应合一报告 | ❌ | ❌ | 字幕+评论双焦点 | 评论链接回字幕的做法可抄 |

### 6.1 竞品留下的空白（= 我们的护城河）

1. **没人把"验真（真/假/可疑）+ 优点/SOP/陷阱萃取 + 溯源"做进同一条视频流水线**。竞品要么只摘要（不验真），要么只验真（不萃取 SOP）。
2. **组合"字幕(核心) + 高光上下文 + 弹幕 + 评论"做内容萃取的极少**；多数把评论只当情感。
3. **针对"一人公司 SOP 自动化"场景**：Scribe 是录自己屏，我们是从**别人教程视频抽可照搬 SOP**——空白。
4. **确定性输出（固定码本 + schema 约束）竞品几乎没提**——我们顺带治"输出不稳定"。

## 7. 内容线处理方案草案（供拍板，未定方案）

### 7.1 内容线内部处理阶段

1. **熟悉化**：整读字幕 + AI 简介（EDA，不编码）。
2. **类型判定**：教程型（抽 SOP）vs 观点型（抽洞察/论点）——内容线第二层分流。
3. **固定码本编码**：核心洞察 / SOP 步骤 / 陷阱 / 事实主张（claim）。
4. **主张验真**：抽 claim → 交 nuwa 三重验证 → 真/假/可疑 + 证据。
5. **组合分析**：高光上下文块（字幕 + 弹幕）深挖；评论做观众侧印证/反驳。
6. **溯源**：每条洞察/主张打时间戳 + 原视频链接。
7. **结构化输出**：Pydantic schema + enum + temperature 0。

> 工程现状提醒：当前 `albedo/watcher/parser.py` 只抽 title/up_name/video_id/正文，**丢弃了互动信号**；`core/models.py` 的 `AlbedoInput.signals` 预留但未填。内容线的组合分析要求 parser 把 Nigredo 中转文件的全部信号读进来——这是要补的缺口。

### 7.2 与 Track B 的接口

- 内容线产出 **DWS-A（内容集市）**，形式线产出 **DWS-B（形式集市）**，ADS 报告两章并列（见 TWOTRACKS）。
- 路由规则：语义路由（学到→A，剪辑→B，没懂→A+B）。

### 7.3 待用户拍板点（5 个）

1. **类型判定**（教程 vs 观点）要不要做自动分流？还是每次都抽全三类（洞察/SOP/陷阱）？
2. **验真范围**：只验"明确事实主张"，还是连"观点/方法有效性"也标可信度？
3. **SOP 抽取颗粒度**：抽到"编号步骤"即可，还是带"安全/质量 callout + 前置条件"的四段式？
4. **组合分析窗口**：高光前后取几秒字幕 + 弹幕（N 值）？
5. **输出稳定性铁律**：是否锁定"固定码本 + schema 约束 + temperature 0"作为内容线标准（顺带治任务二）？
